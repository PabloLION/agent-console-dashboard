# Story: Implement Main Dashboard Layout

**Story ID:** S004.02 **Epic:**
[E004 - TUI Dashboard](../epic/E004-tui-dashboard.md) **Status:** Draft
**Priority:** P1 **Estimated Points:** 5

## Description

As a user, I want to see all my agent sessions displayed in a clear dashboard
layout, So that I can quickly understand the status of each session at a glance.

## Context

The main dashboard layout is the primary view users will see when running the
TUI. It displays all tracked sessions from the daemon with their status, working
directory, and time-since-attention (if applicable). The layout adapts to
different terminal sizes and integrates with the daemon via SUBSCRIBE for
real-time updates.

The dashboard orchestrates widget rendering using the widget system from E005.
This story focuses on the core layout structure, daemon connection, and
real-time update handling. Individual widget implementations are in E005.

All data (sessions and usage) comes from the daemon via SUBSCRIBE. The TUI is a
pure presentation layer that populates `WidgetContext` from daemon broadcasts.
See [widget data flow](../architecture/widget-data-flow.md).

## Implementation Details

### Technical Approach

1. Create `ui.rs` for rendering logic
2. Implement `dashboard.rs` view component
3. Connect to daemon via IPC client using SUBSCRIBE command
4. Receive session and usage data broadcasts from daemon
5. Populate `WidgetContext` with daemon data for widget rendering
6. Render widget output in dashboard layout
7. Display header with title and help hints
8. Show footer with keyboard shortcuts
9. Handle different terminal widths (responsive design)
10. Integrate with `tokio::select!` event loop for real-time updates

### Ratatui Layout Example

```rust
use ratatui::prelude::*;
use ratatui::widgets::{Block, Borders, List, ListItem};

fn render_dashboard(frame: &mut Frame, sessions: &[Session]) {
    let area = frame.area();

    // Create main layout with header, content, footer
    let chunks = Layout::default()
        .direction(Direction::Vertical)
        .constraints([
            Constraint::Length(1),  // Header
            Constraint::Min(3),     // Session list
            Constraint::Length(1),  // Footer
        ])
        .split(area);

    // Render session list
    let items: Vec<ListItem> = sessions
        .iter()
        .map(|s| format_session_line(s, area.width))
        .collect();

    let list = List::new(items)
        .block(Block::default().borders(Borders::ALL).title("Sessions"));

    frame.render_widget(list, chunks[1]);
}
```

### IPC Integration

The dashboard connects to the daemon using SUBSCRIBE to receive real-time
updates. All data (sessions + usage) flows from daemon to TUI:

```rust
// On startup
client.send(Command::Subscribe)?;

// In event loop (tokio::select! arm)
match client.recv() {
    Ok(Message::SessionUpdate(session)) => {
        app.update_session(session);
        // Trigger re-render
    }
    Ok(Message::UsageUpdate(usage)) => {
        app.update_usage(usage);
        // Trigger re-render
    }
    // ...
}
```

### Data Flow

```text
Daemon (SUBSCRIBE broadcasts) → TUI state → WidgetContext → Widgets
```

The TUI never makes external API calls. Daemon fetches usage data from
`claude-usage` crate every 3 minutes and broadcasts to all subscribed TUIs. See
[widget data flow](../architecture/widget-data-flow.md) and
[discussion decisions D3/D4](../architecture/2026-01-31-discussion-decisions.md).
